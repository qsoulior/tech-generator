## 1. Объект испытаний

Объектом испытаний является программное изделие «{{ .program_name }}» версии **{{ .program_version }}**, развёрнутое на стенде `{{ .test_stand }}`. Состав испытываемого комплекса:

- сервис маршрутизации уведомлений;
- сервис расчёта политик доставки;
- адаптеры внешних каналов (push, SMS, email);
- административная панель.

## 2. Цели испытаний

- подтверждение соответствия функциональных требований ТЗ {{ .tz_reference }};
- проверка соответствия показателей надёжности и производительности;
- подтверждение готовности изделия к вводу в промышленную эксплуатацию.

## 3. Требования к программе

- безусловное выполнение всех функций, перечисленных в разделе 4.2 ТЗ;
- работоспособность во всех заявленных режимах;
- корректная обработка нештатных ситуаций (отказ внешнего канала, недоступность БД и т. п.);
- соответствие нормативам безопасности — соответствует РД-ИБ-014.

## 4. Требования к программной документации

К моменту начала испытаний предъявляется:

- настоящая программа и методика испытаний;
- руководство пользователя (ГОСТ 19.505-79);
- описание программы (ГОСТ 19.402-78);
- эксплуатационная документация инфраструктуры стенда.

## 6. Методы испытаний

### 6.1. Функциональные испытания

Для каждой функции (Ф-01…Ф-05 из ТЗ) проводится испытание по схеме: подготовка входных данных → выполнение → сверка фактического результата с ожидаемым.

Пример тест-кейса Ф-02:

```
GIVEN получатель с подпиской на канал push
WHEN  опубликовано событие "order.created"
THEN  через ≤ 1000 мс получатель получает push-уведомление
AND   в журнале доставки появилась запись со статусом delivered
```

### 6.2. Нагрузочные испытания

Профиль нагрузки: ступенчатый рост до целевого RPS с поддержанием в течение 30 минут.

| Показатель | Целевое значение |
| --- | --- |
| Установившийся RPS | {{ .load_target_rps }} |
| Пиковый RPS (5 минут) | {{ .load_peak_rps }} |
| Латентность p95 | ≤ {{ .target_p95_ms }} мс |
| Латентность p99 | ≤ {{ .target_p99_ms }} мс |
| Доля 5xx | ≤ 0,01 % |

### 6.3. Регрессионные испытания

Прогон {{ .regression_tests_count }} тест-кейсов из регресс-набора, включая все ранее обнаруженные дефекты приоритета **major** и выше.

### 6.4. Испытания на безопасность

- статический анализ кода (SAST);
- сканирование контейнеров на CVE;
- проверка ролевой модели — попытки эскалации привилегий;
- проверка хранения и обработки персональных данных.

## 7. Критерии приёмки

- не выявлено критических дефектов (severity = blocker / critical);
- известные дефекты priority = minor допускаются с планом устранения;
- все целевые показатели разделов 6.2 и 6.4 достигнуты.

## 8. Оформление результатов

По итогам испытаний оформляется протокол по форме ПФ-ИСП-01, подписываемый ответственным и утверждающим. К протоколу прилагаются:

- журналы прогонов всех тест-сценариев;
- отчёты k6 (нагрузочные);
- отчёты сканеров безопасности;
- скриншоты дашбордов мониторинга за период испытаний.
